# User Story: Persist Generated Questions in MongoDB

**Story ID:** US-QG-005  
**Epic:** Full-Grade Question Generation & Management  
**Priority:** P0 (Critical)  
**Story Points:** 8  
**Sprint:** TBD

## User Story

```
As a system administrator
I want all AI-generated questions to be stored in MongoDB
So that we can reuse them in the future without calling the LLM every time
```

## Acceptance Criteria

### Functional Requirements

- [ ] **AC-001:** Every question generated by the AI is automatically saved to MongoDB
- [ ] **AC-002:** Stored questions include all metadata: grade, topic, difficulty, format, generation timestamp, and LLM model used
- [ ] **AC-003:** Questions are stored with an initial status of `pending` (not yet approved)
- [ ] **AC-004:** Duplicate detection prevents storing identical questions (same question text + grade + topic)
- [ ] **AC-005:** A `GET /questions` endpoint returns stored questions with filtering by grade, topic, status, and format
- [ ] **AC-006:** Batch generation endpoint allows generating and storing multiple questions at once (e.g. 10 questions per request)

### Technical Requirements

- **REQ-QG-019:** Create a `Question` Mongoose schema with fields: `questionText`, `answer`, `explanation`, `grade`, `topic`, `category`, `format`, `options` (for MCQ), `status` (pending/approved/rejected), `stepByStepSolution`, `metadata`, `createdAt`, `updatedAt`, `reviewedBy`, `reviewedAt`
- **REQ-QG-020:** Create a `QuestionsService` with CRUD operations and filtering
- **REQ-QG-021:** Create a `QuestionsController` for the REST API
- **REQ-QG-022:** Index MongoDB collection on `grade`, `topic`, `status`, and `format` for query performance
- **REQ-QG-023:** Integrate question persistence into the existing generation pipeline

## Dependencies

- US-QG-003 (Grade support for Grades 3–8)
- MongoDB instance running (already configured in `docker-compose.yml`)

## Technical Implementation Notes

- Create `api/src/app/questions/` module with schema, service, and controller
- Use Mongoose `@Schema()` decorator for the entity
- Add compound unique index on (`questionText`, `grade`, `topic`) for deduplication
- Add query indexes on `grade`, `topic`, `status`, `format`
- Integrate with the existing `MathQuestionGeneratorService` to persist after generation
- The `status` field defaults to `pending` — students should only see `approved` questions (enforced in US-QG-006)

## Testing Scenarios

### Scenario 1: Question Stored After Generation

```gherkin
Given I trigger question generation for Grade 5 FRACTION_OPERATIONS
When the AI generates a question
Then the question should be stored in MongoDB
And the status should be "pending"
And all metadata (grade, topic, format, model) should be present
```

### Scenario 2: Duplicate Prevention

```gherkin
Given a question "What is 15 + 8?" for Grade 3 ADDITION already exists
When the AI generates the same question text for Grade 3 ADDITION
Then the duplicate should not be stored
And the system should log a duplicate detection event
```

### Scenario 3: Query Stored Questions

```gherkin
Given there are 50 stored questions across grades 3-8
When I query GET /questions?grade=5&topic=FRACTION_OPERATIONS&status=approved
Then I should receive only Grade 5 FRACTION_OPERATIONS questions with approved status
```

### Scenario 4: Batch Generation

```gherkin
Given I request batch generation of 10 questions for Grade 4 MULTIPLICATION
When the generation completes
Then all 10 questions should be stored in MongoDB
And each should have status "pending"
```
